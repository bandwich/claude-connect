# Slash Command Behavior Reference

Empirically tested on Claude Code 2.1.83 (2026-03-25).

## Transcript Write Patterns

Every command that writes to the transcript follows this structure:

```
Line N:   role=user  <local-command-caveat>Caveat: The messages below were generated by the user while running local commands. DO NOT respond to these messages or otherwise consider them in your response unless the user explicitly asks you to.</local-command-caveat>
Line N+1: role=user  <command-name>/cmd</command-name>\n<command-message>cmd</command-message>\n<command-args>args</command-args>
Line N+2: role=user  <local-command-stdout>output text here</local-command-stdout>  (optional â€” not all commands produce stdout)
```

Additional lines may appear: `role=system` (empty metadata), `role=file-history-snapshot` (empty).

## Command Categories

### Category 1: Writes caveat + command + stdout

These write all three XML-tagged lines. The stdout contains the command's output (sometimes with ANSI escape codes).

| Command | stdout content |
|---------|---------------|
| `/context` | Context usage bar with ANSI colors |
| `/effort` | `Effort level: auto (currently medium)` |
| `/plan` | `Enabled plan mode` / `Disabled plan mode` |
| `/compact` | `[2mCompacted (ctrl+o to see full summary)[22m` (ANSI) |

`/compact` also prepends a summary message (`role=user`, plain text starting with "This session is being continued from a previous conversation...") before the caveat block. It **appends** to the existing transcript â€” does NOT rewrite or truncate.

### Category 2: Writes caveat + command, no stdout

These write the caveat and command-name lines but no `<local-command-stdout>` line. The actual output is terminal-only.

| Command | Terminal behavior |
|---------|------------------|
| `/cost` | Inline text, auto-dismisses |
| `/status` | Tabbed UI overlay, Esc to dismiss |

After `/status` is dismissed, a `âŽż Status dialog dismissed` line appears in the terminal but the dismiss text is NOT written to the transcript.

### Category 3: Writes nothing to transcript

Zero lines added. Output is a terminal overlay requiring user interaction to dismiss.

| Command | Terminal behavior | Dismiss mechanism |
|---------|------------------|-------------------|
| `/btw` | Overlay with response text | Press Space, Enter, or Escape |
| `/help` | Scrollable command list | Esc |
| `/fast` | Toggle selector | Tab to toggle, Enter to confirm, Esc to cancel |
| `/model` | Model selector list | Enter to confirm, Esc to exit |
| `/config` | Searchable settings UI | Type to filter, Esc to clear |
| `/diff` | Interactive diff viewer | Esc |

### Category 4: Skills (expand into prompts)

Skills are defined in `.claude/skills/*/SKILL.md` files. When invoked, they expand into user prompt messages that Claude processes normally. These produce standard `role=user` and `role=assistant` transcript entries â€” no XML tags.

| Command | Source |
|---------|--------|
| `/commit`, `/debug`, `/simplify`, `/loop`, `/claude-api`, `/batch` | Bundled skills |
| `/init` | Builtin (triggers Claude to create CLAUDE.md) |
| Any custom skill | `~/.claude/skills/` or `<project>/.claude/skills/` |

Unknown skills produce: caveat + plain text `Unknown skill: name`.

## /clear Behavior

`/clear` creates a **new transcript file** (new session ID). The old file is left untouched.

New file contents:
```
Line 0: role=file-history-snapshot (empty)
Line 1: role=user  <local-command-caveat>...</local-command-caveat>
Line 2: role=user  <command-name>/clear</command-name>...
Line 3: role=system (empty)
```

Subsequent messages are written to the **new** file. The transcript watcher must detect the file switch or it will keep watching the old (now-stale) file.

## /compact Behavior

`/compact` **appends** to the existing transcript file. It does NOT rewrite or truncate. Line count increases. No watcher issues with file identity â€” only needs to process the new lines normally.

## Terminal Overlay Interaction

Category 3 commands produce interactive terminal overlays that require keypresses to interact with or dismiss. The specific keypresses needed per command:

- **Space/Enter/Escape to dismiss**: `/btw`
- **Esc to dismiss/exit**: `/help`, `/diff`, `/status`, `/model`
- **Tab to toggle + Enter to confirm + Esc to cancel**: `/fast`
- **Type to filter + Enter/arrows to select + Esc to clear**: `/config`

The server already has `tmux send-keys` capability via `TmuxController` which can send any of these keypresses. The pane parser (`pane_parser.py`) can read the terminal output via `tmux capture-pane`.

## Impact on Voice Server

### Current problems

1. **Category 3 commands produce no visible response in the app** â€” the app shows "Thinking" briefly then nothing, because there's no transcript output to stream. The terminal overlay sits there blocking the session until someone manually dismisses it.

2. **`<local-command-caveat>` and `<local-command-stdout>` messages are sent to iOS as raw user text** â€” the transcript watcher doesn't filter or transform these XML-tagged messages. They appear as ugly raw XML in the conversation view.

3. **`/clear` breaks the transcript watcher** â€” after `/clear`, a new transcript file is created but the watcher continues monitoring the old file. No new messages are detected.

4. **Delivery verification fails for category 3 commands** â€” `verify_delivery` looks for the sent text in the transcript as a user message. Category 3 commands never appear in the transcript, so voice-sent commands report "delivery failed."

### Existing infrastructure that can help

- **`tmux capture-pane`**: Can read exactly what's displayed in the terminal, including overlay content. The pane parser already does this every 1s for activity detection.
- **`tmux send-keys`**: Can send Space, Enter, Escape, Tab to dismiss overlays or interact with selectors. Already used for sending user input.
- **`pane_parser.py`**: Already parses terminal output for activity states. Could be extended to detect and extract overlay content.

### Key files

- `voice_server/handlers/input_handler.py` â€” sends text to tmux, verifies delivery via transcript
- `voice_server/services/transcript_watcher.py` â€” watches transcript JSONL, tracks `processed_line_count`
- `voice_server/infra/pane_parser.py` â€” parses tmux pane for activity state
- `voice_server/infra/tmux_controller.py` â€” `send_input()` and `send_escape()` for tmux interaction
- `voice_server/services/commands_provider.py` â€” builds command list from builtins + skills
